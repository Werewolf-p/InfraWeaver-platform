The Game Hub is InfraWeaver Console’s opinionated control surface for game servers running on Kubernetes. It wraps deployments, PVCs, DNS helpers, console streaming, metrics, and access control into one workflow so you can operate a server without touching raw manifests.

## What the Game Hub is

A Game Hub server is usually backed by:

- a Deployment in the `game-hub` namespace
- one or more Services for game traffic
- a PVC for persistent world data
- a ConfigMap containing egg metadata and quick commands
- annotations used by DNS helpers, metrics panels, and audit flows

## Creating a new server

Open **Game Hub** and click **New Server**. The wizard intentionally walks you through the same order the platform expects you to reason about a deployment.

> **Screenshot callout:** On the server creation page, the left side shows the current step while the main panel changes between egg selection, configuration, resource sliders, and final review.

### Step 1 — Browse eggs

Choose between:

- **Built-in eggs** for curated defaults maintained inside the repo
- **Pelican eggs** fetched from the remote catalog for wider game coverage

Use search and category filters to narrow the list.

### Step 2 — Configure

Required environment variables are surfaced as friendly form fields. InfraWeaver pre-populates defaults from the egg so you start from a known-good baseline.

> **Screenshot callout:** The configure step shows the selected egg summary at the top, then generated fields such as server name, world name, EULA, passwords, and gameplay options below it.

### Step 3 — Resources

Pick memory, CPU, storage size, storage class, and DNS hostname. Capacity warnings come from cluster-aware checks so you can see when a deployment would stress the busiest node.

> **Screenshot callout:** The resources step includes sliders, current requested values, and a warning panel if projected memory pressure is too high.

### Step 4 — Deploy

Review the final payload, then deploy. The console creates the PVC, Deployment, Service, and ConfigMap in sequence.

> **Screenshot callout:** The final review page highlights the selected egg, resource plan, generated hostname, and environment variables before the deploy button is enabled.

## What Pelican eggs are and why they exist

Pelican eggs are reusable templates borrowed from the Pelican/Pterodactyl ecosystem. They standardize:

- Docker image defaults
- startup and stop commands
- environment variables
- default ports
- quick console commands

They exist so the platform can support many games without rewriting deployment logic for each one.

## Managing servers after deployment

Open any server from the Game Hub list to reach the detail view.

### Start, stop, and restart

The action bar issues lifecycle operations against the underlying Deployment or Pod. Use:

- **Start** when the server is intentionally scaled down or idle
- **Stop** before maintenance, snapshots, or edits to sensitive files
- **Restart** after configuration changes that require a fresh process

## Using the Console tab

The Console tab streams logs and lets you send commands to the game server process.

Common workflows include:

- sending `save-all`, `players`, `status`, or `help`
- reviewing recent console history for joins, leaves, warnings, and crashes
- using quick commands generated by the selected egg

The console keeps a local history depth selector so operators can switch between a short live stream and several days of retained context.

## CPU and RAM metrics charts

InfraWeaver pulls time-series data from Prometheus. The charts generally show:

- CPU usage versus CPU limit
- memory working set versus memory limit
- rolling history for the currently selected pod

This is useful for spotting:

- sustained throttling
- memory leaks or runaway modpacks
- under-sized or over-sized resource requests

## Files tab

The Files tab exposes the server’s mounted data path so you can:

- browse directories
- open and edit text files
- upload assets or configuration files
- remove stale files after a backup or migration

Treat file edits as production changes. Save a world or stop the server before editing critical configuration.

## DNS quick-add

From the server page you can create a hostname that points at the server without opening `/dns` separately. This is ideal when you want a memorable name such as `minecraft.games.int.rlservers.com` instead of sharing a raw node address and port.

## RBAC for server access

Access is scoped. A user might be able to:

- view one server only
- operate an entire class of servers under `/game-hub/`
- administer every game server on the platform

Server-level sharing is managed from the Users page or the access controls inside the server detail view.
